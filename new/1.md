I'd make one small refinement to the earlier design.

Instead of letting the executor own the queue, the `JobManager` should own it. The reason is that the standard `ExecutorService` doesn't let you efficiently remove an arbitrary queued task by job ID. Owning the queue makes `cancel(jobId)` straightforward.

The implementation below uses:

* one worker thread
* one `LinkedBlockingQueue`
* one currently-running job
* FIFO scheduling
* cooperative cancellation
* thread-safe APIs
* automatic removal of finished jobs

---

## BackgroundJob.java

package jobs;

public interface BackgroundJob extends Runnable {

    /**
     * Called by the JobManager when cancellation is requested.
     * Implementations should stop as soon as practical.
     */

}

## JobDescriptor.java

```java
package jobs;

import java.time.Instant;
import java.util.UUID;

public final class JobDescriptor {

    private final UUID id;
    private final String name;
    private final String details;

    private volatile JobStatus status;

    private final Instant submittedAt;
    private volatile Instant startedAt;
    private volatile Instant finishedAt;

    private volatile Throwable failure;

    JobDescriptor(UUID id,
                  String name,
                  String details) {

        this.id = id;
        this.name = name;
        this.details = details;

        this.status = JobStatus.QUEUED;
        this.submittedAt = Instant.now();
    }

    JobDescriptor(JobDescriptor other) {
        this.id = other.id;
        this.name = other.name;
        this.details = other.details;
        this.status = other.status;
        this.submittedAt = other.submittedAt;
        this.startedAt = other.startedAt;
        this.finishedAt = other.finishedAt;
        this.failure = other.failure;
    }

    public UUID id() {
        return id;
    }

    public String name() {
        return name;
    }

    public String details() {
        return details;
    }

    public JobStatus status() {
        return status;
    }

    public Instant submittedAt() {
        return submittedAt;
    }

    public Instant startedAt() {
        return startedAt;
    }

    public Instant finishedAt() {
        return finishedAt;
    }

    public Throwable failure() {
        return failure;
    }

    void status(JobStatus status) {
        this.status = status;
    }

    void started() {
        this.startedAt = Instant.now();
        this.status = JobStatus.RUNNING;
    }

    void cancelled() {
        this.finishedAt = Instant.now();
        this.status = JobStatus.CANCELLED;
    }

    void failed(Throwable t) {
        this.finishedAt = Instant.now();
        this.failure = t;
        this.status = JobStatus.FAILED;
    }
}
```

## I would make two improvements before using this in production

### 1. Add a `COMPLETED` status

Currently, finished jobs simply disappear because you only wanted active jobs returned. Internally, though, having

```java
COMPLETED
```

makes the lifecycle explicit and easier to debug.

### 2. Replace `requestCancellation()` with `cancel()`

This is a naming preference, but I think

```java
job.cancel();
```

reads better than

```java
job.requestCancellation();
```

while still conveying cooperative cancellation.

Overall, this design is simple, fully thread-safe, uses only the JDK, guarantees FIFO execution with exactly one running job, supports cancellation of queued and running jobs, and is well suited for a JavaFX application.
